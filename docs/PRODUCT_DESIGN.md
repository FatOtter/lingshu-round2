## 1. 产品概述

LingShu 是以 Ontology 为核心的数据操作系统。用户通过本体模块建模、数据模块关联和探索数据、能力模块执行操作、智能体模块用 AI 完成上述一切。

用户有两种方式完成同一操作：通过各模块的 GUI 界面点击操作，或通过智能体用自然语言完成。两种方式调用相同的后端能力，产出一致的结果。

## 2. 用户角色与核心场景

### 2.1 角色定义

角色描述的是**职责**，不是人。一个用户可以同时承担多个角色。RBAC 权限按角色授予（详见 `SETTING_DESIGN.md` §2.5）。

| 角色 | 核心职责 | 主要活动区域 | 典型背景 |
|------|---------|-------------|---------|
| Schema 设计者 | 定义实体/关系模型，管理版本发布，配置数据映射 | 本体（主）、数据（验证映射） | 领域专家、数据架构师 |
| 数据工程师 | 接入数据源，验证数据质量，管理数据分支 | 数据（主）、本体（查看定义） | 数据工程、ETL 开发 |
| 能力开发者 | 开发 Global Function 和 Workflow，调试 Action 执行 | 能力（主）、本体（查看 ActionType）、数据（验证结果） | 后端开发、低代码开发 |
| 业务用户 | 通过自然语言完成数据查询、操作执行、报表获取 | 智能体（主）、数据（浏览） | 业务分析师、运营、管理层 |
| 系统管理员 | 管理用户/租户/权限，审查审计日志 | 设置（主） | IT 管理 |

**业务用户是最大的用户群体**。Copilot 的核心目标是让业务用户无需理解底层模型和技术概念，通过对话完成日常工作。其他四个角色负责"建设系统"，业务用户负责"使用系统"。

### 2.2 核心场景

以下场景覆盖系统的主要使用链路。每个场景标注涉及的角色和模块，描述 GUI 路径和 Copilot 路径两条线。

#### 场景 1：从零建模到数据可用

**角色**：Schema 设计者 → 数据工程师

```
阶段 1：建模（Schema 设计者）
  本体模块 → 新建 ObjectType（如"设备"）→ 添加 PropertyType → 保存 Draft
  → 新建 LinkType（如"设备-所属-车间"）→ 配置 source/target
  → 版本管理 → Submit to Staging → Review → Publish

阶段 2：映射数据源（Schema 设计者 + 数据工程师）
  数据模块 → 新建数据源连接 → 填写连接信息 → 测试连接
  本体模块 → ObjectType 编辑器 → 数据映射 Tab → 配置 AssetMapping（read_connection_id + read_asset_path）
  → 重新发布版本

阶段 3：验证（数据工程师）
  数据模块 → 数据浏览 → 选择"设备"→ 查看实例列表 → 确认字段映射正确
```

**Copilot 路径**：`"帮我创建一个设备类型，包含名称、编号、状态三个属性"` → Copilot 调用 Ontology Action 创建类型 → `"连接到 PostgreSQL 数据库的 equipment 表"` → Copilot 调用数据源配置和映射。

---

#### 场景 2：定义操作并执行

**角色**：Schema 设计者 → 能力开发者 → 业务用户

```
阶段 1：定义 ActionType（Schema 设计者）
  本体模块 → 新建 ActionType（如"更新设备状态"）
  → 配置参数（derived_from: 设备实例引用 + 目标状态枚举）
  → 配置 implementation（引擎类型、outputs 声明）→ 发布

阶段 2：调试执行（能力开发者）
  能力模块 → 原子能力列表 → 找到"更新设备状态"→ 进入执行页
  → 填写参数（选择设备实例 + 选择状态）→ 执行 → 查看结果
  → 数据模块 → 验证实例已更新

阶段 3：业务使用（业务用户）
  智能体模块 → "把 3 号产线的设备状态改为维护中"
  → Copilot 解析意图 → 查询设备实例 → 展示 ConfirmationCard
  → 用户确认 → 执行完成 → 展示结果
```

---

#### 场景 3：工作流编排与自动化

**角色**：能力开发者 → 业务用户

```
阶段 1：编排工作流（能力开发者）
  能力模块 → 新建工作流 → DAG 编辑器
  → 添加节点："查询异常设备"（builtin Function）
  → 添加节点："批量更新状态"（Action）
  → 添加节点："发送通知"（webhook Function）
  → 连线 → 配置条件分支 → 保存

阶段 2：手动执行验证（能力开发者）
  能力模块 → 工作流列表 → 执行 → 填写触发参数 → 逐步确认 → 查看执行历史

阶段 3：Copilot 触发（业务用户）
  智能体模块 → "运行一下异常设备处理流程"
  → Copilot 匹配工作流 → 展示参数确认 → 执行 → 流式展示各步骤结果
```

---

#### 场景 4：数据探索与分析

**角色**：业务用户（主）、数据工程师

```
GUI 路径：
  数据模块 → 数据浏览 → 选择"设备"→ 筛选（状态 = 运行中）→ 排序（创建时间 ↓）
  → 点击实例 → 查看详情 → 关系面板查看关联的"车间""维护记录"
  → 点击操作链接 → 跳转能力模块执行 Action

Copilot 路径：
  "运行中的设备有多少？按车间统计"
  → Copilot 调用 query_instances → 聚合计算 → 返回 MetricCard + Table
  "3 号车间的设备最近一周有维护记录吗？"
  → Copilot 调用 get_related_links → 时间过滤 → 返回 Table
```

---

#### 场景 5：多租户与权限管理

**角色**：系统管理员

```
首次部署：
  系统启动 → Seed 创建默认租户 + admin 用户 → admin 登录（/login）
  → 设置模块 → 修改初始密码

日常管理：
  设置模块 → 用户管理 → 创建用户 → 分配角色（admin/member/viewer）
  → 租户管理 → 创建新租户 → 添加成员
  → 审计日志 → 按模块/用户/时间筛选 → 点击资源 RID 跳转到对应模块查看详情

租户切换（任何用户）：
  Global Header → 用户菜单 → 切换租户 → 系统签发新 JWT → 页面刷新，数据切换到目标租户
```

---

#### 场景 6：Copilot 辅助建模（跨角色）

**角色**：Schema 设计者 + Copilot Shell

```
Schema 设计者在本体模块编辑 ObjectType
→ 打开 Copilot Shell → "这个类型应该包含哪些常见属性？"
→ Copilot 基于类型名称和描述推荐属性列表
→ 用户选择采纳 → Copilot 调用 Action 批量添加 PropertyType

切换到数据映射 Tab → "帮我把这些属性映射到 equipment 表的字段"
→ Copilot 查询数据源 schema → 自动匹配 → 生成映射建议
→ 用户确认 → 写入 AssetMapping
```

> **Copilot Shell vs 智能体模块**：Shell 感知当前页面上下文（正在编辑的实体、当前模块），回答和操作聚焦于上下文内。智能体模块的对话是全局的，适合跨模块任务和开放式探索。

### 2.3 场景与模块的关联矩阵

| 场景 | 本体 | 数据 | 能力 | 智能体 | 设置 |
|------|:----:|:----:|:----:|:------:|:----:|

| 3. 工作流编排与自动化 | | | ★ | ○ | |

## 3. 整体页面结构

### 3.1 全局布局

```
┌──────────────────────────────────────────────────────────────────┐
│  Global Header (50px)                                            │
│  [Logo] [面包屑]        [全局搜索]        [Copilot 开关] [头像]  │
├──────┬──────────────────────────────────────────┬────────────────┤
│ Dock │              Main Stage                  │   Copilot      │
│(64px)│                                          │   Shell        │
│      │  ┌──────────┬───────────────────────┐    │  (400px,       │
│  ◇   │  │ 侧边面板  │       内容区域        │    │   可拖拽,      │
│  ▦   │  │ (250px,  │                       │    │   默认收起)    │
│  ⚡   │  │  可折叠)  │                       │    │               │
│  🤖   │  │          │                       │    │               │
│  ⚙   │  └──────────┴───────────────────────┘    │               │
│      │                                          │               │
└──────┴──────────────────────────────────────────┴────────────────┘
```

系统采用**三栏弹性布局**：

| 区域 | 宽度 | 说明 |
|------|------|------|
| Dock | 64px 固定 | 一级导航，图标 + 微文字 |
| Main Stage | 自适应 | 核心工作区，内部统一为**侧边面板 + 内容区域**两栏结构 |
| Copilot Shell | 400px 默认，可拖拽至 80% 屏宽 | 全局 AI 辅助面板，默认收起 |

- **Main Stage** 骨架统一：左侧侧边面板（250px，可折叠）用于模块内导航，右侧内容区域展示具体页面
- **Copilot Shell** 是全局组件，不属于任何模块。在本体/数据/能力/设置模块中可随时展开，获得 AI 辅助。切换到智能体模块时，Shell 自动收起且禁止打开——智能体模块的 Main Stage 本身即为完整的 AI 交互界面，避免两个对话区域并存

### 3.2 Global Header

| 区域 | 内容 |
|------|------|
| 左 | Logo（点击返回首页）+ 面包屑（`模块 > 分区 > 实体名称`） |

**用户头像下拉菜单**：当前用户信息、租户切换、个人设置、退出登录。

### 3.3 一级导航（Dock）

| 图标 | 模块 | 路由前缀 |
|------|------|---------|
| ◇ | 本体 | `/ontology` |
| ▦ | 数据 | `/data` |

## 4. 首页

路由：`/`，点击 Logo 返回。

首页提供系统概览和快捷入口，帮助用户快速了解系统状态并跳转到常用功能。具体内容和布局在首页设计中定义，大致包含：

- 系统数据概览（Ontology 实体统计、数据实例规模）

- 快捷操作入口
- 系统动态（版本发布、操作执行记录）

## 5. 模块设计

以下描述每个模块的定位、侧边面板分区和主要页面。各模块的详细交互设计在各自的模块设计文档中定义。

### 5.1 本体模块

**定位**：管理 Ontology 类型定义、数据映射和版本。

**侧边面板**：

| 分区 | 说明 |
|------|------|
| 概览 | Ontology 全局拓扑图 |
| 对象类型 | ObjectType 管理 |
| 关系类型 | LinkType 管理 |
| 接口类型 | InterfaceType 管理 |
| 动作类型 | ActionType 管理 |
| 共享属性类型 | SharedPropertyType 管理 |
| 属性类型 | 跨所有实体的 PropertyType 扁平索引 |
| 数据映射 | AssetMapping 配置总览 |
| 版本管理 | Draft → Staging → Snapshot → Active |

点击分区进入对应的列表页或管理页。点击列表中的具体实体进入编辑器，侧边面板同步展示当前实体的关联节点用于快速导航。

**主要页面类型**：
- 列表页：各实体类型的列表（搜索、筛选、新建）
- 编辑器：实体的分 Tab 编辑（属性、逻辑、数据映射等，各实体类型 Tab 不同）
- 版本管理：Diff 对比、提交、发布、回滚
- 数据映射总览：所有 ObjectType / LinkType 的映射状态，快速进入配置

**发布后约束**：已发布实体的部分字段不可修改（API Name、数据类型等），视觉上以灰色背景 + 锁图标标识。

### 5.2 数据模块

**定位**：管理数据源连接，浏览和探索 Ontology 类型关联的数据实例，管理数据版本。

**侧边面板**：

| 分区 | 说明 |
|------|------|
| 概览 | 数据模块全局概览（数据源状态、实例规模、分支状态） |
| 数据源 | 数据源连接的配置和管理（连接信息、状态监控） |
| 数据浏览 | 按 ObjectType / LinkType 浏览数据实例 |
| 数据版本 | 数据分支管理（创建分支、切换分支、合并） |

**主要页面类型**：
- 数据源管理：数据源连接列表（名称、类型、状态）、连接配置表单、连接测试
- 类型选择：选择要浏览的 ObjectType 或 LinkType（只展示已配置 AssetMapping 的类型）
- 实例列表：选定类型的实例列表（列由 PropertyType 动态生成，标准表格渲染）
- 实例详情：ObjectType 实例或 LinkType 实例的详情（属性面板根据 WidgetConfig 渲染、关系面板、操作历史、相关操作链接跳转到能力模块）
- 分支管理：创建/切换/合并数据分支，查看分支间差异

**数据源与 AssetMapping 的关系**：AssetMapping 在本体模块中定义读写路径（`read_connection_id` + `read_asset_path`），其中 `connection_id` 引用的数据源连接在数据模块中配置和管理。

### 5.3 能力模块

**定位**：管理和执行系统中所有可调用能力，包括原子能力（Ontology Action + Global Function）和由原子能力编排而成的工作流。

**侧边面板**：

| 分区 | 说明 |
|------|------|
| 概览 | 能力模块全局概览（能力统计、最近执行记录） |
| 原子能力 | Ontology Action 和 Global Function 的统一浏览与执行 |
| 工作流 | 将原子能力编排为工作流，管理和执行 |

**主要页面类型**：
- 原子能力列表：统一展示 Action 和 Global Function，按类型分组
- Action 执行：参数表单（根据 ActionType 动态生成）、执行按钮、执行历史。Action 的定义编辑跳转到本体模块
- Global Function 管理：函数注册、配置、参数定义、版本管理、执行测试
- 工作流列表：已创建的工作流
- 工作流编辑：编排原子能力，定义执行顺序、条件分支、错误处理
- 工作流执行：参数填写、执行、执行历史和状态追踪

### 5.4 智能体模块

**定位**：通过自然语言与 AI 智能体交互完成任务，管理智能体的能力配置和基础设施。

**与 Copilot Shell 的区别**：Copilot Shell 以当前模块的上下文为核心，聚焦于模块内的辅助操作，可拒绝无关请求。智能体模块的对话是通用的，可调用系统集成的所有能力。

**侧边面板**：

| 分区 | 说明 |
|------|------|
| 对话 | 通用对话界面（模块默认页） |
| 会话 | 历史会话管理 |
| 基座模型 | 可用 LLM 模型的配置和管理 |
| Skill | 技能注册和管理 |
| MCP | MCP 服务连接管理 |
| Sub-Agent | 子智能体配置和管理 |
| 监控 | 执行追踪和推理链查看 |

**主要页面类型**：
- 对话界面：与智能体的通用对话（模块默认页，消息流中包含文本和 A2UI 结构化组件）
- 会话列表：历史会话，按时间分组，支持搜索和恢复
- 基座模型管理：模型列表、模型配置（API 连接、参数调整）
- Skill 管理：技能列表、技能注册和配置
- MCP 管理：MCP 服务列表、连接配置和状态
- Sub-Agent 管理：子智能体列表、创建和配置（系统提示词、工具绑定、安全策略）
- 执行追踪：推理链可视化、工具调用记录、耗时分析

**A2UI**：智能体通过 A2UI 协议流式生成结构化 UI 组件（Table、MetricCard、Form、ConfirmationCard 等）。A2UI 的组件规范和协议细节在智能体模块设计文档中定义。

### 5.5 设置模块

**定位**：系统管理，包括认证、用户管理、审计日志聚合查询、租户管理。详见 `SETTING_DESIGN.md`。

**侧边面板**：

| 分区 | 说明 |
|------|------|
| 概览 | 用户/租户统计、最近审计活动 |
| 用户管理 | 用户 CRUD、状态管理、角色分配 |
| 审计日志 | 跨模块审计日志统一查询（按模块/用户/时间/事件筛选） |
| 租户管理 | 租户 CRUD、成员管理 |

**主要页面类型**：
- 登录页（`/login`，认证布局外，不显示 Dock 和 Global Header）
- 用户列表 + 详情/编辑页
- 审计日志表格（支持跨模块资源跳转）
- 租户列表 + 详情（含成员管理）
- 概览：统计卡片 + 最近审计活动

## 6. 通用交互原则

以下原则贯穿所有模块，确保一致的用户体验。各原则的具体实现方式在各模块设计中细化。

### 6.1 列表页

- 所有列表页提供搜索、筛选、排序、分页能力
- 新建操作统一在列表页左上角提供入口
- 批量操作通过勾选多行触发

### 6.2 编辑与保存

- 表单直接在页面内编辑，修改后字段出现变更指示
- 离开未保存页面时弹出确认弹窗
- 不可编辑字段以灰色背景 + 锁图标标识

### 6.3 删除与确认

按风险分级处理：
- 低风险：直接执行，Toast 提示
- 中风险：确认弹窗
- 高风险：确认弹窗 + 影响分析 + 输入确认
- 有依赖：阻止操作，展示引用列表

### 6.4 加载与空状态

- 加载中：骨架屏或 Spinner
- 空数据：引导文案 + 操作按钮
- 加载失败：错误提示 + 重试按钮

### 6.5 反馈与通知

- 操作成功：Toast，自动消失
- 操作失败：Toast + 错误详情，手动关闭
- 校验错误：字段内联提示
- 系统通知：页面顶部 Banner

### 6.6 跨模块导航

**原则：引用即链接**。凡是展示了实体名称或 RID 的地方，都应可点击跳转到对应的编辑器或详情页，无论目标在哪个模块。

典型场景包括但不限于：本体模块内各实体间的相互引用跳转、数据实例详情跳转到能力模块执行 Action、能力模块跳转到本体模块查看 ActionType 定义、版本管理 Diff 跳转到对应实体编辑器等。

完整的跳转路径在各模块设计文档中枚举。
